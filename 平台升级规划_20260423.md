> 规划时间：2026年4月23日

## 一、核心目标（用户定义）

### 1. 副业避坑
- 真实的数据（平台运营状态、收入数据、用户评价）
- 真实的案例（成功/失败，每种至少100条）
- 真实的骗局预警（骗局类型、手法、案例、投诉）
- 真实的投诉数据（用户投诉、处理进度、曝光）

### 2. 帮助用户寻找副业/创业

- 案例参考（真实案例，可筛选、可搜索）
- 商业逻辑（每个副业方向的商业模式分析）
- 时效性保证（定期更新，标注最后验证时间）

### 3. 功能真实有效
- 不是入口摆设，每个功能都能用、有用
- 数据可验证、可追溯
- 用户体验流畅无BUG

### 4. 用户积累与变现
- 开通账户体系（注册、登录、个人中心）
- 用户行为数据收集
- 变现路径：会员、广告、平台合作、数据服务

### 5. 数据后台
- 用户数据统计（访问量、注册量、活跃度）
- 内容数据管理（案例、平台、骗局增删改）
- 运营数据监控（转化率、留存率）

## 三、技术架构升级

### 当前架构（静态网站）
```
用户 → GitHub Pages → 静态HTML/CSS/JS → JSON数据文件
```

### 目标架构（动态平台）
```
用户 → 前端(Vue/React) → API服务 → 数据库
                    ↓
              管理后台 → 数据管理
                    ↓
              定时任务 → 数据更新
```

### 技术栈选择

#### 方案A：全栈自建（推荐）
- **前端**：Vue 3 + Vite（或继续用静态HTML+JS）
- **后端**：Node.js + Express 或 Python + FastAPI
- **数据库**：PostgreSQL 或 MongoDB
- **部署**：Vercel + Supabase（免费起步）或 云服务器

#### 方案B：无代码/低代码平台
- **前端**：Webflow / Bubble
- **后端**：Supabase / Firebase
- **优点**：快速上线，成本低
- **缺点**：定制性受限

#### 方案C：静态+第三方服务
- **前端**：继续用静态HTML
- **后端服务**：Supabase（数据库+认证+API）
- **优点**：改动最小，保留现有代码
- **缺点**：功能受限

## 七、关键决策点

### 决策1：技术架构选择
- **选项A**：全栈自建（推荐，灵活可控）
- **选项B**：Supabase + 静态前端（快速，改动小）
- **选项C**：低代码平台（最快，受限）

**需要确认**：你倾向哪个方案？

### 决策2：数据收集方式
- **选项A**：自己收集整理（质量可控，慢）
- **选项B**：众包/外包收集（快，需审核）
- **选项C**：用户投稿为主（慢，需激励）

**需要确认**：怎么收集数据？

### 决策3：上线节奏
- **选项A**：数据完善后再上后端（稳妥）
- **选项B**：后端和数据并行（快）
- **选项C**：先上后端，数据逐步补充（激进）

**需要确认**：你希望怎么推进？
